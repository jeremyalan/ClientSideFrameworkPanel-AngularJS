Common Scenarios

  • URL Routing

    Pro Tip: Always use ui-router, DO NOT USE built-in ng-view 
    $stateProvider.state('app', {
   abstract: true,
   url: '',
   template: '''
     <div class="container">
       <div class="row">
         <div class="col-xs-3">
           <div ui-view="sidebar"></div>
         </div>
         <div class="col-xs-9">
           <div ui-view="main"></div>
         </div>
       </div>
     </div>
   '''

$stateProvider.state('app.home', {
  url: '/home',
  views: {
    'sidebar': {
      controller: 'SidebarCtrl',
      templateUrl: 'templates/sidebar.html',
    },
    'main': {
      controller: 'MainCtrl',
      templateUrl: 'templates/main.html'
    }
  }
});

  • Data Access

    • Use built-in or custom services to make calls to backend or load data from client-side storage
    • Several options available:
      ngResource Built by Angular team, available as custom module 'ngResource'. Useful when working with RESTful services.
      Restangular Community plugin, similar to ngResource but provides more flexibility and customization.
      $http Low-level HTTP library provided by Angular Core. Not necessarily recommended, prefer to wrap usage with custom services.
      Custom Services We use a code-generation template to convert our Web API interfaces in .NET to Angular services.

  • Data Binding

    • Strong separation between Controller and View
    • $scope allows different features (controllers, views, directives) to share data
    • Two-way data binding via HTML expressions or wired up manually via $scope.$watch

  • Third-party plugins

    • Most jQuery plugins can be wrapped in a directive
    • Sometimes, it's easier and more performant to re-implement plugins in Angular

  • Documentation

    • Available at https://docs.angularjs.org/api
    • Generated via js-doc code documentation, so always up-to-date and relevant.
    • API Reference Guide, not always great for beginners, usually helps to already know what you're looking for.
    • Some high-level concepts are documented, but usually better just to go to the web and read articles published by the community.
